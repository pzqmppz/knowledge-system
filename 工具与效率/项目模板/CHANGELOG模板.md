# CHANGELOG.md 变更日志模板

> 轻量级变更记录，解决文档漂移问题

---

## 📋 使用说明

1. **只记录变更，不重写文档**
2. **标注变更影响**：是否需要更新文档
3. **Claude 优先读取此文件**，了解哪些文档已过期
4. **每次变更都记录**，哪怕很小的改动

---

## 📝 模板格式

```markdown
# [项目名称] 变更日志

> 所有重要变更都将记录在此文件

## 格式说明
- [新增] - 新功能
- [变更] - 功能变更
- [修复] - Bug 修复
- [废弃] - 功能废弃
- [文档] - 文档更新

---

## [未发布]

### [新增]
- 暂无

### [变更]
- 暂无

### [修复]
- 暂无

### [废弃]
- 暂无

### [文档]
- 暂无

---

## [1.0.0] - 2026-03-02

### [新增]
- 用户认证功能
- 数据库连接池

### [文档]
- 初始化架构文档
```

## 🔄 与文档同步

变更记录时，标注对文档的影响：

```markdown
## [0.6.1] - 2026-03-02

### [新增]
- 添加用户认证模块 `src/auth/`

### [变更]
- 用户 API 从 `/api/user` 改为 `/api/users`

### [影响]
- ⚠️ docs/ARCHITECTURE.md 需要更新（新增认证模块）
- ⚠️ docs/API.md 需要更新（API 路径变更）
- ✅ 无需更新配置文件

### [同步状态]
- [ ] docs/ARCHITECTURE.md 已同步
- [ ] docs/API.md 已同步
```

## 📊 文档同步状态表

在文件末尾维护一个同步状态表：

```markdown
---

## 📊 文档同步状态

| 文档 | 最后更新 | 当前代码版本 | 同步状态 | 备注 |
|------|----------|--------------|----------|------|
| CLAUDE.md | 2026-03-02 | v0.6.3 | ✅ 同步 | 始终保持最新 |
| docs/ARCHITECTURE.md | 2026-02-28 | v0.6.3 | ⚠️ 滞后 | 缺少认证模块 |
| docs/API.md | 2026-03-01 | v0.6.3 | ⚠️ 滞后 | API 路径已变更 |
| docs/DEPLOYMENT.md | 2026-02-15 | v0.5.0 | ✅ 同步 | 部署流程未变 |

### 图例
- ✅ 同步 - 文档与代码一致
- ⚠️ 滞后 - 文档需要更新
- ❌ 废弃 - 文档已过时，不再适用
```

## 🤖 Claude 使用此文件的方式

```yaml
工作流程:
  1. Claude 首先读取 CHANGELOG.md 了解最新变更
  2. 检查"文档同步状态表"，识别过时文档
  3. 遇到标记为 ⚠️ 的文档时，优先去代码中验证
  4. 如果用户询问过时功能，Claude 提醒用户文档可能过期
  5. 完成变更后，Claude 提醒用户更新 CHANGELOG

示例对话:
  用户: "项目的用户 API 是什么？"
  Claude: "根据 CHANGELOG，用户 API 已从 /api/user 变更为
          /api/users（v0.6.1），docs/API.md 可能还未同步。
          让我查看代码确认..."
```

## 💡 最佳实践

1. **频繁更新** - 每次变更都记录，不用等到版本发布
2. **标注影响** - 明确说明哪些文档需要更新
3. **使用复选框** - 用 `[ ]` 和 `[x]` 跟踪同步进度
4. **保持简洁** - 不写长篇大论，只记录关键信息
5. **定期回顾** - 每周清理一次，更新同步状态
