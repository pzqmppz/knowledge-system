## ⚡ 模式 B：加速模式（质量门控增强）

**适用：** 熟练开发者、提高效率

**特点：** 质量门控自动检查、写时代码质量强制

**预计对话：** 15-20 轮/功能

**效率提升：** 比学习模式节省 30% 时间

---

## 工作流程

```
┌─────────────────────────────────────────────┐
│  加速模式微循环                               │
├─────────────────────────────────────────────┤
│ 对话 1: "/tdd [功能描述]"                    │
│         ↓                                   │
│         (自动完成 Red + Green)               │
│         ↓                                   │
│      /plankton-code-quality                 │
│         (自动格式化 + lint)                  │
├─────────────────────────────────────────────┤
│ 对话 2: "/quality-gate"                     │
│         ↓                                   │
│      (自动检查：代码质量 + 安全 + 测试)      │
│         ↓                                   │
│         → 如果有 CRITICAL：自动修复          │
│         → 如果全部通过：继续                 │
├─────────────────────────────────────────────┤
│ 对话 3: "/verify quick"                     │
│         ↓                                   │
│         → 如果通过：人工测试                 │
├─────────────────────────────────────────────┤
│ 对话 4: (人工验收)                          │
├─────────────────────────────────────────────┤
│ 对话 5: "/checkpoint create"                │
└─────────────────────────────────────────────┘

总计：3-4 轮对话，比学习模式节省 30% 时间
```

## 加速模式 vs 学习模式

| 维度 | 学习模式 | 加速模式 | 提升 |
|------|---------|---------|------|
| 对话轮数 | 40-50 轮/功能 | 15-20 轮/功能 | **60% ↓** |
| 人工介入 | 每步确认 | 只需关键决策 | **70% ↓** |
| 质量保证 | 手动审查 | 自动门控 | **更稳定** |
| 学习效果 | 深度理解 | 效率优先 | **-** |
| 适用场景 | 新手学习 | 熟练开发 | **-** |

---

## 加速模式的优势

1. **效率大幅提升**：60% 对话轮数减少
2. **质量自动保证**：质量门控实时监控
3. **写时代码质量**：plankton 自动格式化和 lint
4. **减少人为错误**：自动化检查减少遗漏
5. **技术债管理**：建议自动记录，不阻塞开发

## 加速模式的局限

1. **学习效果较弱**：自动化流程不利于理解原理
2. **依赖配置完善**：需要完整配置质量门控
3. **灵活性降低**：严格按照流程执行

## 质量门控配置

### 基础配置

```yaml
# .claude/settings.yaml
qualityGate:
  enabled: true
  metrics:
    testCoverage:
      target: 80
      blocker: true
    codeComplexity:
      target: 10
      blocker: true
    codeDuplication:
      target: 5
      blocker: false
    typescriptErrors:
      target: 0
      blocker: true
    eslintWarnings:
      target: 0
      blocker: false
```

## 最佳实践

1. **信任自动化**：让工具做它擅长的事
2. **关注 CRITICAL**：只处理阻塞问题
3. **技术债管理**：建议记录到清单，定期回顾
4. **定期审查**：每周回顾技术债清单
5. **持续优化**：根据项目情况调整质量标准

---

> **💡 使用建议：**
>
> **什么时候使用加速模式**：
> ✅ 已熟悉 TDD 流程
> ✅ 项目时间紧张
> ✅ 需要稳定的质量保证
> ✅ 团队协作开发
>
> **什么时候切换到自动模式**：
> ⚠️ 功能清晰且重复性高
> ⚠️ 需要进一步减少人工介入
