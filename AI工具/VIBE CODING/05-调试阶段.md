## 🔵 阶段 5：调试阶段 (Debug)

**目标：** 快速定位和修复问题

**触发时机：** 当开发或验证过程中出现问题时

**预计时间：** 根据问题复杂度，5-30 分钟

**特点：** 可随时进入，不限于特定阶段

## Step 2：问题定位和修复

```markdown
【对话 D2】
你："定位问题：搜索结果不显示"

Claude:
##### 问题定位流程

## 🔍 步骤 1：确认问题
```typescript
// 在 REPL 中测试
> const { results, loading } = useSearch('/api/search')
> console.log(results)
[]  // 结果为空
```

## 🔍 步骤 2：检查 API
```typescript
> // 直接调用 API
> const response = await fetch('/api/search?q=test')
> const data = await response.json()
> console.log(data)
{ results: [] }  // API 返回空
```

## 🔍 步骤 3：检查后端
```bash
##### 在后端 REPL 中
> console.log(process.env.SEARCH_API_URL)
undefined  // 环境变量未设置！
```

## 🎯 问题根因

**环境变量 `SEARCH_API_URL` 未配置**

## 🔧 修复方案

##### 1. 添加环境变量
```bash
###### .env.local
SEARCH_API_URL=https://api.example.com/search
```

##### 2. 验证修复
```typescript
> // 再次测试
> const data = await fetch('/api/search?q=test')
> console.log(await data.json())
{ results: [{ id: '1', title: 'Test' }] }  // ✅ 有结果了！
```


> **💡 使用建议：**
>
> **问题定位技巧**：
> 1. 先确认问题（是什么）
> 2. 再检查输入（输入是否正确）
> 3. 然后检查处理（逻辑是否正确）
> 4. 最后检查输出（输出是否符合预期）
>
> **常见问题**：
> - 环境变量未设置 → 检查 .env
> - API 调用失败 → 检查网络和权限
> - 类型错误 → 检查 TypeScript 配置
> - 测试失败 → 检查 mock 数据
```

### 问题定位详解

#### 定位方法论

```markdown
##### 1. 5 Whys 分析法

**问题**：搜索结果不显示

**Why 1**：为什么结果不显示？
→ 因为 results 数组为空

**Why 2**：为什么数组为空？
→ 因为 API 返回空数组

**Why 3**：为什么 API 返回空？
→ 因为后端调用失败

**Why 4**：为什么调用失败？
→ 因为环境变量未设置

**Why 5**：为什么环境变量未设置？
→ 因为 .env.local 文件不存在

**根因**：缺少环境变量配置
**修复**：创建 .env.local 文件

##### 2. 二分定位法

**问题**：页面加载慢

**定位过程**：
1. 检查前端：加载时间 2s ✅
2. 检查 API：响应时间 5s ❌
3. 检查后端：处理时间 4s ❌
4. 检查数据库：查询时间 3.5s ❌

**根因**：数据库查询慢
**修复**：优化查询或添加索引

##### 3. 对比法

**问题**：测试失败

**对比过程**：
1. 对比通过的测试：输入、环境、配置
2. 对比失败的测试：输入、环境、配置
3. 发现差异：失败的测试缺少 mock 数据

**根因**：缺少 mock 数据
**修复**：补充 mock 数据
```

#### 常见问题模式

```markdown
##### 模式 1：环境问题

**症状**：
- 功能本地正常，部署后失败
- 环境变量读取失败
- 依赖版本不匹配

**诊断**：
```bash
> // 检查环境变量
> console.log(process.env.MY_VAR)
undefined

> // 检查依赖版本
> npm list package-name
```

**修复**：
- 检查 .env 文件
- 检查 package.json
- 检查部署配置

##### 模式 2：异步问题

**症状**：
- 数据未更新
- 状态不同步
- 竞态条件

**诊断**：
```typescript
> // 检查 Promise 状态
> const promise = asyncFunction()
> console.log(promise)
Promise { <pending> }

> // 等待完成
> await promise
```

**修复**：
- 正确使用 async/await
- 添加适当的等待
- 处理并发

##### 模式 3：类型问题

**症状**：
- TypeScript 错误
- 运行时类型错误
- 类型推断失败

**诊断**：
```typescript
> // 检查类型
> typeof variable
'object'

> // 检查实例
> variable instanceof MyClass
true
```

**修复**：
- 添加类型注解
- 使用类型守卫
- 修复类型定义

##### 模式 4：状态问题

**症状**：
- 状态不更新
- 旧状态保留
- 状态不同步

**诊断**：
```typescript
> // 检查状态
> console.log(state)
{ value: 'old' }

> // 触发更新
> setState('new')

> // 检查更新
> setTimeout(() => console.log(state), 0)
{ value: 'new' }
```

**修复**：
- 正确使用 setState
- 检查依赖数组
- 处理异步更新
```

### 回滚和恢复详解

#### Checkpoint 策略

```markdown
##### 策略 1：软回滚 + 修复

**适用场景**：问题可以快速修复

**步骤**：
```bash
# 1. 保存当前工作
$ git stash save "WIP: debugging search issue"

# 2. 回滚到稳定点
$ git reset --soft phase-2-complete

# 3. 修复问题
# ... 修复代码 ...

# 4. 重新提交
$ git add .
$ git commit -m "fix: search API configuration"

# 5. 恢复其他更改
$ git stash pop
```

**优点**：
- 保留所有更改
- 可以选择性恢复
- 安全可靠

**缺点**：
- 需要手动管理 stash

##### 策略 2：硬回滚 + 重做

**适用场景**：问题复杂，重新开始更快

**步骤**：
```bash
# 1. 丢弃所有更改
$ git reset --hard phase-2-complete

# 2. 重新执行
你："重新执行 Phase 3，注意 API 配置"

# 3. 生成新代码
# ... 新代码 ...
```

**优点**：
- 干净的状态
- 重新生成代码
- 避免累积问题

**缺点**：
- 丢失当前更改
- 需要重新开发

##### 策略 3：分支隔离

**适用场景**：尝试多种解决方案

**步骤**：
```bash
# 1. 创建调试分支
$ git checkout -b debug/search-issue

# 2. 尝试修复方案 A
# ... 修复代码 ...

# 3. 如果失败，回滚
$ git reset --hard HEAD~1

# 4. 尝试修复方案 B
# ... 修复代码 ...

# 5. 成功后合并
$ git checkout main
$ git merge debug/search-issue
```

**优点**：
- 可以尝试多种方案
- 不影响主线
- 保留所有尝试

**缺点**：
- 需要管理多个分支
```
