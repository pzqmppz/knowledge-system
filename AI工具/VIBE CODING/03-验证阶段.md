## 🔴 阶段 3：验证阶段 (Verification)

**目标：** 确保代码符合生产标准

**预计时间：** 30-45 分钟

**触发条件：** 所有执行阶段 Phase 完成

---

## 验证阶段详细说明

### 质量门控详解

#### 代码质量指标

| 指标类别 | 指标名称 | 目标值 | 权重 | 计算方式 |
|---------|---------|--------|------|---------|
| 测试覆盖 | 语句覆盖率 | >80% | 30% | 已执行语句数 / 总语句数 |
| 测试覆盖 | 分支覆盖率 | >75% | 20% | 已执行分支数 / 总分支数 |
| 测试覆盖 | 函数覆盖率 | 100% | 10% | 已调用函数数 / 总函数数 |
| 代码质量 | 圈复杂度 | <10 | 15% | 每个函数的平均复杂度 |
| 代码质量 | 代码重复率 | <5% | 10% | 重复代码行数 / 总代码行数 |
| 类型安全 | TypeScript 错误 | 0 | 10% | 类型错误数量 |
| 代码规范 | ESLint 警告 | 0 | 5% | Lint 警告数量 |

**综合评分计算**：
```
质量分数 = Σ(指标得分 × 权重)
优秀：≥90 分
良好：≥80 分
合格：≥70 分
不合格：<70 分
```

## 验证失败处理

### 处理流程

```
验证失败
    ↓
分析失败原因
    ↓
    ├─ 代码问题 → 回到开发阶段修复
    ├─ 配置问题 → 调整配置后重新验证
    ├─ 测试问题 → 修复测试后重新验证
    └─ 环境问题 → 修复环境后重新验证
            ↓
        重新验证
            ↓
        验证通过 → 继续
```

## 最佳实践

1. **逐步验证**：按顺序执行验证步骤
2. **详细记录**：记录所有验证结果
3. **及时修复**：发现问题立即修复
4. **自动化**：尽可能自动化验证流程
5. **持续改进**：基于验证结果优化流程
