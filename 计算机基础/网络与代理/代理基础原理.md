# 代理基础原理

> 理解代理是如何"接管"你的网络流量的

## 一、为什么需要代理

当你访问互联网时，数据包的路径是：

```
你的电脑 → 本地网络 → ISP → 目标网站服务器
```

这个路径中，**每一环都能看到你的真实 IP**，而且某些网站可能限制访问。

代理的作用是插入一个"中间人"：

```
你的电脑 → 代理服务器 → 目标网站服务器
```

目标网站看到的是代理服务器的 IP，而不是你的。

## 二、代理的两种接管方式

### 1. 系统代理（System Proxy）

```
┌─────────────────────────────────────────────────────┐
│                      应用程序                        │
│                          │                          │
│                          ▼                          │
│               检查系统代理设置                        │
│              (HTTP_PROXY / HTTPS_PROXY)             │
│                          │                          │
│            ┌─────────────┴─────────────┐            │
│            ▼                           ▼            │
│    有代理设置                    无代理设置          │
│        │                           │               │
│        ▼                           ▼               │
│   发送到代理                   直接连接目标          │
└─────────────────────────────────────────────────────┘
```

**特点：**
- ✅ 轻量，不需要额外权限
- ✅ 只代理支持代理设置的应用（主要是浏览器）
- ❌ **不代理所有流量**（比如命令行工具、游戏、某些系统服务）
- ❌ UDP 流量通常不被代理

### 2. TUN 模式（虚拟网卡）

```
┌─────────────────────────────────────────────────────┐
│                      应用程序                        │
│                          │                          │
│                          ▼                          │
│                   操作系统网络栈                      │
│                          │                          │
│                          ▼                          │
│                    路由表判断                         │
│                          │                          │
│            ┌─────────────┴─────────────┐            │
│            ▼                           ▼            │
│    匹配 TUN 路由               不匹配 TUN 路由       │
│        │                           │               │
│        ▼                           ▼               │
│   发送到 TUN 虚拟网卡          发送到物理网卡         │
│        │                                           │
│        ▼                                           │
│   Clash 处理（根据规则分流）                         │
└─────────────────────────────────────────────────────┘
```

**特点：**
- ✅ **接管所有流量**（TCP + UDP）
- ✅ 对应用程序透明，不需要配置代理
- ✅ 可以代理游戏、命令行、系统更新等
- ❌ 需要**管理员权限**
- ❌ 可能与某些 VPN 软件冲突
- ❌ 配置不当会导致**所有流量都走代理**（包括本该直连的）

### 你的情况：TUN 模式的问题

你使用 Clash Verge + TUN 模式，问题可能出在：

```
问题场景：
你访问 baidu.com
    ↓
TUN 接管流量
    ↓
Clash 根据规则判断
    ↓
规则没匹配到 "baidu.com 直连"
    ↓
流量走了代理服务器
    ↓
代理服务器在日本/新加坡/美国
    ↓
百度的服务器看到请求来自海外 IP
    ↓
百度把响应发给海外 CDN 节点
    ↓
你收到的响应很慢
```

## 三、代理协议简介

| 协议 | 特点 | 适用场景 |
|------|------|----------|
| HTTP | 明文，最基础 | 简单的网页浏览 |
| HTTPS | 加密，可过 CDN | 需要伪装流量 |
| SOCKS5 | 支持 TCP+UDP | 游戏、P2P |
| Shadowsocks | 轻量加密 | 个人使用 |
| VMess | V2Ray 原生协议 | 功能丰富 |
| Trojan | 伪装成 HTTPS | 抗检测 |
| Hysteria2 | 基于 QUIC，高速 | 高延迟网络 |

Clash 支持多种协议，你配置文件里的 `proxies` 部分就是定义这些代理服务器。

## 四、Clash 的工作流程

```
                    ┌─────────────────┐
                    │   入站流量       │
                    │ (TUN/系统代理)   │
                    └────────┬────────┘
                             │
                             ▼
                    ┌─────────────────┐
                    │   DNS 解析       │ ← 这里很容易出问题！
                    │ (域名 → IP)      │
                    └────────┬────────┘
                             │
                             ▼
                    ┌─────────────────┐
                    │   规则匹配       │
                    │ (按顺序匹配规则)  │
                    └────────┬────────┘
                             │
          ┌──────────────────┼──────────────────┐
          ▼                  ▼                  ▼
    ┌───────────┐     ┌───────────┐     ┌───────────┐
    │  DIRECT   │     │  REJECT   │     │  PROXY    │
    │  (直连)    │     │  (拒绝)    │     │  (代理)    │
    └───────────┘     └───────────┘     └───────────┘
```

### 关键点

1. **DNS 解析先于规则匹配** - 如果 DNS 解析出了问题，后面的规则再完美也没用
2. **规则按顺序匹配** - 第一条匹配的规则生效，后面的不再检查
3. **规则类型决定匹配方式** - DOMAIN-SUFFIX 只匹配域名后缀，IP-CIDR 匹配 IP 段

## 🎯 核心要点回顾

1. **TUN 模式接管所有流量**，这既是优势也是风险
2. **DNS 解析先于规则匹配**，DNS 配置错误会导致规则失效
3. **规则按顺序匹配**，顺序很重要
4. **问题往往是多层的**：DNS + 规则 + 网站检测同时作用
