## 五、经验教训

### 1. 过度依赖 AI 自动化的陷阱

**问题**：一开始执着于"完全通过口述 + API 完成所有操作"，拒绝任何手动操作 Coze 工作流平台。

**反思**：
- [[Coze]] 是非开源平台，官方文档更新不及时，AI 对其理解有限
- 如果一开始让 AI 教我一步一步手动操作，反而能更快熟悉平台
- [[工作流]] 编排很难一次成功，经常有参数问题，需要结合 API 调试

**收获**：最终摸索出了一套工作流编排方法，并将其沉淀为 [[Skill]] 能力，为后续重复性工作提供保障。

---

### 2. 功能拆解不够原子化

**问题**：B 端开发时，只描述了一级、二级功能，三级功能全交给 AI 自行设计，导致：
- 部分细小功能被遗漏
- 代码开发后没有与前端联调
- 测试时反复发现功能缺失

**反思**：每个功能点必须拆解到原子级别，明确验收标准，不能笼统地交给 AI。

---

### 3. 部署方案的前期规划不足

**问题**：
- 最初使用海外免费平台部署 + 海外免费数据库
- 上线前发现 [[Coze]] 平台无法访问后端 API
- 临时决定迁移到腾讯云，但域名审核未完成

**反思**：
- 不能过度追求"免费方案"，前期需要根据场景设计评估潜在风险
- 涉及国内服务调用的项目，必须考虑网络可达性
- 厌恶损失心理会阻碍问题的快速解决

---

### 4. [[AI]] 协作的边界把控

**问题**：当 AI 遇到错误时，经常会自行搜索网页，采取一些不合时宜的替代方案，而不是先询问用户。

**反思**：
- AI 不是全知全能，会犯错，会"自作聪明"
- 需要在提示词中增加约束：遇到计划外情况时，必须先向用户确认
- 当发现 AI 开始做"奇怪的方案"时，要立即打断并纠正

---

### 5. 前端设计能力的短板

**问题**：
- 即使使用了 [[Skill]] 工具，前端效果仍只是"将就能用"的水平
- 发现问题但无法用语言准确描述（比如"这里有问题，但说不出来是什么问题"）
- 需要截图丢给多模态模型，让它告诉我该怎么描述

**改进方向**：
- 测试更强大的前端 [[AI]] 协助方案
- 系统学习前端组件知识，知道它们叫什么、有什么样式
- 提升用语言描述 UI 问题的能力

## 八、总结

这次项目是在极短时间内完成的 [[MVP]] 产品。虽然过程中走了不少弯路，但也积累了宝贵的实战经验：

1. **工具选择要务实**：不要为了"全自动化"而拒绝更高效的手动方案
2. **需求拆解要细致**：原子化的功能点才能保证交付质量
3. **部署方案要前瞻**：考虑网络环境、合规性等现实因素
4. **[[AI]] 协作要有边界**：建立约束机制，避免 AI 自作主张
5. **短板要主动补**：前端设计、技术表达能力需要系统提升

这台云服务器后续可以做更多项目尝试，发挥更大价值。
