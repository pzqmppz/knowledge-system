## 技术总结

### 核心技术

| 技术点 | 实现方案 |
|--------|----------|
| **前端框架** | Next.js 14 App Router + React 18 |
| **后端** | Next.js API Routes + TypeScript |
| **AI 处理** | GLM-4.7-Flash / DeepSeek-V3 双模型 |
| **知识存储** | Notion API (结构化数据库) |
| **缓存/锁** | Upstash Redis (分布式状态) |
| **日志** | Pino (结构化 JSON 日志) |
| **监控** | Prometheus (实时指标) |
| **文档解析** | jszip (EPUB), pdf-parse (PDF), mobi-parser |

### 核心功能

1. **多格式文档解析**
   - EPUB：完全支持，多层回退策略
   - MOBI/AZW3：标准格式支持
   - PDF：字体大小检测章节
   - 网页：智能去除广告

2. **AI 逻辑研磨器**
   - 去水存骨：提取逻辑骨架
   - 餐巾纸时刻：抽象为核心图形
   - 跨界迁移：自动思考应用场景
   - 稳定性评估：区分时效与基石知识

3. **自动分级与评分**
   - 相关性评分：1-10 分
   - 行动优先级：立即执行 vs 参考备用

## 遇到的问题

| # | 问题 | 影响 | 解决方案 |
|---|------|------|----------|
| 1 | EPUB 内容仅提取 10 字符 | 核心功能不可用 | 多层提取策略 |
| 2 | Vercel 函数超时限制 | 无法处理长文档 | Docker 本地部署 |
| 3 | 前后端状态不同步 | 用户体验差 | 后端返回 pageId，前端轮询 |
| 4 | 仪表盘数据不更新 | 功能可信度下降 | 直接查询 Notion 数据库 |
| 5 | AI 请求超时 | 大文件处理失败 | 超时 120s→240s |
| 6 | Notion API 超时误判 | 错误标记"源失效" | 区分错误类型 |

## 经验教训

### 做得好的地方

1. **多层回退策略** - EPUB 解析采用 xmlMode → 正则兜底
2. **混合部署模式** - Vercel 前端 + Docker 后端
3. **可观测性完善** - 日志 + 指标 + 健康检查
4. **错误处理系统化** - 重试、降级、熔断器

### 需要改进的地方

1. **测试覆盖率** - 单元测试不足，目标 60%
2. **异步架构** - 未实现消息队列，仍是同步处理
3. **流式处理** - 大文件内存占用较高
4. **格式支持** - DOCX、TXT 未支持

### 关键经验

#### 设计原则
1. 永远假设输入会出错
2. 了解你的环境限制（Serverless ≠ 无限）
3. I/O 密集型操作要并发
4. 错误处理要系统化
5. API 依赖要显式化
6. 可观测性是必需品

#### 防御性设计
```typescript
// 永远不要信任输入
const validated = schema.parse(body);

// 假设所有外部调用都会失败
const result = await withRetry({
  fn: () => ai.process(content),
  maxRetries: 3,
  timeout: 65000,
});

// 使用熔断器
const breaker = new CircuitBreaker('ai-service', {
  failureThreshold: 5,
  recoveryTimeout: 60000,
});
```
