---

## 一、产品概述

### 1.1 核心功能
用户输入想学习的主题，AI自动生成完整的课程体系，包括：
- 分章节的课程内容
- 双人对话式音频播客
- 结构化学习指南
- 结业测验

### 1.2 产品形态
```
用户输入问题 → 流式输出内容 → 分章节展示 → Next Section翻页 → 章节末测验
```

## 三、各模块设计分析

### 3.1 课程内容（Course Content）

#### 结构特点
| 元素 | 设计 | 作用 |
|------|------|------|
| 章节标题 | 清晰的层级结构 | 建立知识框架 |
| 概念解释 | 技术性描述 | 精确传递知识 |
| 代码示例 | 可运行的代码片段 | 实践参考 |
| 对比表格 | 多维度对比 | 权衡取舍 |
| 引用标注 | 带来源链接 | 可追溯、增加可信度 |
| 章节测验 | 选择题 | 检验理解 |

#### 引用标注示例
```
"Extracting information from academic PDF documents is crucial..."

— A Benchmark of PDF Information Extraction Tools Using a Multi-Task
  and Multi-Domain Evaluation Framework for Academic Documents
  arXiv: https://arxiv.org/abs/2303.09957
```

### 3.2 音频播客（Podcast）

#### 核心发现：音频不是文字的朗读，而是「重新演绎」

| 维度 | 文字（课本） | 音频（对话） |
|------|------------|------------|
| **目的** | 精确传递知识 | 让知识变得可亲近 |
| **语言** | 技术性、书面 | 口语化、有情感 |
| **结构** | 线性罗列 | 问答递进 |
| **独特内容** | 代码、引用、表格 | 个人经历、吐槽、惊叹 |
| **缺失内容** | 无 | 代码示例、引用来源 |

#### 双人角色分工

| 角色 | 功能 | 话术特点 |
|------|------|---------|
| **SPEAKER_00（主持人）** | 提问、共情、总结、吐槽 | "Tell me about it" / "Whoa" / "Sounds like..." |
| **SPEAKER_01（专家）** | 解释、深入、引导、肯定 | "Exactly" / "Think of it as..." / "You're thinking like an engineer" |

**关键设计**：主持人代表"学习者视角"，专家代表"知识权威"

#### 对话递进结构

```
[00:00] 开场引入
├── 主持人：设场景 "feeding the beast"
└── 专家：定调 "most projects fall apart here"
     ↓
[00:21] 痛点共鸣
├── 主持人：分享失败经历 "word salad"
└── 专家：解释原因 "蛇形读取"
     ↓
[00:54] 解决方案1 (LlamaParse)
├── 专家：引入工具
├── 主持人：追问 "有什么不同"
└── 专家：解释 + 主持人确认理解
     ↓
[01:48] 解决方案2 (JSON)
├── 主持人：继续问 "还有什么"
├── 专家：解释JSON模式
└── 主持人：幽默总结 "控制狂"
     ↓
[02:16] 解决方案3 (GROBID)
├── 专家：引入另一个工具
├── 主持人：吐槽 "像青蛙" ← 情感点缀
└── 专家：解释功能
     ↓
[02:48] 综合应用
├── 主持人：提出组合方案
└── 专家：肯定 "你思考得像个工程师"
     ↓
[03:09] 升级挑战 (Boss Level)
├── 专家：悬念 "最难的部分"
├── 主持人：猜测
└── 专家：揭晓 "表格和图片"
     ↓
[03:34] 终极方案 (VLM)
├── 专家：引入视觉模型
└── 主持人：惊叹 "科幻感"
     ↓
[04:28] 总结收尾
└── "Garbage in, garbage out" 金句
```

#### 对话递进模式总结

```
痛点引入 → 个人经历共鸣 → 解决方案1 → 追问确认 → 解决方案2 →
幽默总结 → 解决方案3 → 情感点缀 → 综合应用 → 升级挑战 →
终极方案 → 惊叹收尾 → 金句总结
```

### 3.3 学习指南（Study Guide）

#### 六种结构化模板

| 模块 | 内容 | 学习价值 |
|------|------|---------|
| **📖 Core Concepts** | 核心概念概述 | 建立知识框架 |
| **📌 Must Remember** | 按主题分类的必记要点 | 快速回顾核心 |
| **📚 Key Terms** | 术语定义+使用场景+主题 | 不只解释，还告诉你在哪用 |
| **🔍 Key Comparisons** | 表格对比不同方案 | 清晰展示权衡取舍 |
| **⚠️ Common Mistakes** | 错误→原因→正确做法 | 从反面学习，避免踩坑 |
| **🔄 Key Processes** | 步骤分解的关键流程 | 理解执行顺序 |

#### Common Mistakes 设计亮点
```
❌ 错误做法：对复杂的多栏学术论文使用简单的文本提取器（如 PyPDF2）

为什么会这样：这是最简单、最常见的方法，但它对文档布局"视而不见"

✅ 正确做法：使用结构感知的解析器（如 LlamaParse 或 GROBID）
           正确读取分栏、表格和标题，防止下游出现乱码文本
```

#### Key Comparisons 设计亮点
```
特性            | 固定大小分块           | 语义分块
----------------|------------------------|---------------------------
切分逻辑        | 按固定字符数           | 按概念相似度
上下文感知      | 低。经常切断思路       | 高。找到自然断点
实现难度        | 简单快速               | 较复杂，需要模型
最佳场景        | 通用文本               | 学术论文、法律文档

记忆口诀：固定大小 = 快但脆弱
         语义分块 = 聪明、结构化
```

## 五、产品形态适用性分析

### 5.1 适合场景：引导型学习

| 特征 | 说明 |
|------|------|
| 有明确终点 | 学习完就有收获 |
| 需要流程化 | 步骤清晰，循序渐进 |
| 知识结构化 | 可以被组织成章节 |
| 适合测验 | 有正确答案可验证 |

### 5.2 不适合场景：探索型研究

| 特征 | 说明 |
|------|------|
| 没有固定路径 | 研究者需要自由探索 |
| 发现意外关联 | 比按部就班更重要 |
| 开放式问题 | 没有标准答案 |
