### 核心知识/价值点

#### 技术维度

**知识点清单**：

**1. AI 工程的核心差异**

| 维度 | 传统 ML 工程 | AI 工程（基于基础模型） |
|------|-------------|---------------------|
| 数据需求 | 大量标注数据 | 少量样本或零样本 |
| 训练需求 | 从零训练模型 | 使用预训练模型 |
| 开发重心 | 特征工程、模型选择 | Prompt 工程、系统设计 |
| 技能要求 | 统计学、算法 | 评估、安全、应用架构 |

**2. 语言模型训练的两个阶段**

```
预训练（Pre-training）
   ↓
   目标：预测下一个词
   ↓
   数据：任何自然文本
   ↓
   优化目标：熵/困惑度（Perplexity）
   ↓
   结果：拥有世界知识的统计模型

后训练（Post-training）
   ↓
   目标：对齐人类偏好
   ↓
   技术：SFT、RLHF、DPO
   ↓
   优化目标：有用性、安全性
   ↓
   结果：ChatGPT/Gemini/Claude 的差异化
```

**3. 采样策略与温度参数**

```
温度 = 0
   ↓
贪婪采样：总是选择最可能的 token
   ↓
结果：可预测、稳定、"无聊"

温度 = 1
   ↓
标准采样：按原始概率分布
   ↓
结果：平衡的多样性和质量

温度 > 1
   ↓
高温采样：倾向于低概率 token
   ↓
结果：创造性、可能不连贯
```

**关键洞察**：
- 采样是无需重新训练的"廉价优化"
- 调试时检查概率分布可发现模型不确定性
- 创造力与连贯性需要权衡

**4. 评估驱动开发（Evaluation-Driven Development）**

```
业务问题定义
   ↓
设计业务指标（时间、成本、满意度）
   ↓
映射到模型指标（准确率、F1、相关性）
   ↓
构建离线评估系统
   ↓
生产环境持续监控
```

**为什么评估是最大瓶颈**：
- 推荐系统：可直接测量 CTR/购买率
- 代码生成：可编译、可测试
- 文本生成：主观性高，难以量化
- 大多数企业用例缺乏清晰的 ROI 指标

**5. AI as a Judge 的问题**

| 维度 | 优势 | 挑战 |
|------|------|------|
| 相关性 | 与人类判断高度相关 | 依赖具体模型和提示词 |
| 成本 | 比人工评估便宜 | 不够稳定（非平稳性） |
| 速度 | 实时反馈 | 缺乏标准化定义 |
| 控制权 | 易于部署 | 下游团队可能不知道提示词 |

**6. In-Context Learning**

```
零样本学习（Zero-shot）
   ↓
   指令：判断这是否是垃圾邮件
   ↓
   输入：[邮件内容]
   ↓
   模型直接输出判断

少样本学习（Few-shot）
   ↓
   示例1：[邮件A] → 垃圾邮件
   示例2：[邮件B] → 正常邮件
   示例3：[邮件C] → 正常邮件
   ↓
   输入：[新邮件]
   ↓
   模型从示例中学习模式并判断
```

**为什么重要**：
- 无需重新训练模型
- 使通用模型适应特定任务
- 语言模型"万能"的关键

**7. RAG vs 长上下文迷思**

```
常见误解：长上下文会让 RAG 过时

反驳理由1：需求无限增长
   手机内存变大 → 数据中心消失？
   上下文变长 → 我们总用完所有空间

反驳理由2：容量 ≠ 效率
   Claude/GPT-4 支持百万 token
   但超过 10,000 token，模型就"混乱"
   Chip 的小说写作经历验证了这一点

结论：RAG 仍然必要
   - 精确检索相关内容
   - 降低处理负担
   - 提高响应质量
```

**8. AI Agent 的规划难题**

```
Agent 定义（基于 Russell & Norvig）
   ↓
感知环境 + 交互环境
   ↓
传感器（信息输入）+ 执行器（行动输出）

Agent 的两个核心组件
   ↓
1. 工具使用（Tool Use）
   - 已相对成熟（Function Calling）

2. 规划（Planning）
   - 仍然是巨大挑战
```

**规划为什么难**：
- 需要预测行动结果
- 搜索空间指数级增长
- 人类最优计划 ≠ AI 最优计划
- 难以生成训练数据

**Yang LeCun 的观点**：自回归 LLM 不能规划

**Chip 的观点**：可能只是缺乏合适的工具和足够强的模型

**关键洞察**：

1. **"后训练是模型差异化的秘密 sauce"**
   - 预训练：大家都用同样的互联网数据
   - 后训练：数据策展、人类偏好收集方式不同
   - 这解释了为什么各主流模型有不同"个性"

2. **"评估能力决定 AI 采纳速度"**
   - 不是最先进的技术先普及
   - 是最容易评估的用例先普及
   - 推荐系统、代码生成、欺诈检测

3. **"Prompt 工程是写作，不是编程"**
   - 任何人都能写，但很少有人能写好
   - 需要系统化方法，而非"技巧"堆砌
   - 版本控制和实验追踪是关键

4. **"长上下文不会杀死 RAG"**
   - 就像笔记本内存不会让数据中心消失
   - 我们总会用完所有可用资源
   - 效率问题比容量问题更根本

**核心洞见提炼**：

4. **"Agent 现状的诚实评估"**
   - 工具使用已经成熟
   - 规划仍然是巨大挑战
   - 不要过度炒作 Agent 能力

**启发**：这是一个很棒的类比反驳。当我们说"长上下文会让 RAG 过时"时，就像说"大内存会让数据中心过时"。现实是：无论内存多大，我们总会找到用完它的方式。更重要的是：容量不等于效率。100 万 token 的上下文窗口不意味着模型能"理解"这 100 万 token。Chip 的亲身经历（小说写作）很有说服力——超过 10,000 token，模型就开始胡说八道。RAG 的价值不仅是"塞更多内容"，而是"精确找到需要的内容"。

### 可复用资产

#### 金句

1. "Evaluation is the biggest bottleneck for AI adoption."
   - 评估是 AI 采纳的最大瓶颈。

2. "As a more intelligent AI becomes, the harder it is to evaluate it."
   - AI 越智能，评估它就越困难。

3. "A lot of the failures are silent."
   - 很多失败是无声的。

4. "Prompt engineering is very easy to get started. But like to do so effectively does require a lot of practices and a lot of discipline to do so like systematically."
   - Prompt 工程很容易上手，但要有效做到需要大量练习和系统化方法。

5. "Anyone can do it, but not many people can do so effectively."
   - 任何人都能做，但很少有人能做好。

6. "I do think that sampling is very cool. It's a cheap way to improve the model, the application's performance, without having to retrain the model."
   - 采样非常酷。它是一种无需重新训练就能提高模型和应用性能的廉价方式。

7. "No matter how big my phone memory or my laptop memory is, I will always run out of memory."
   - 无论我的手机或笔记本内存有多大，我总会用完。

8. "Just because a model can fit in a million token context, doesn't mean that it can process that million token efficiently."
   - 模型能容纳百万 token 上下文，并不意味着它能高效处理这百万 token。

9. "Planning is very, very, very hard."
