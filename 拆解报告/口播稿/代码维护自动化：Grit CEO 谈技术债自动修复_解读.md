## 叙事骨架

### 开篇钩子
**核心洞察**：GitHub Copilot 帮你写新代码，但世界上 90% 的代码是已经写好的、需要维护的旧代码。

**问题引入**：开发者 1/3 的时间花在维护、迁移、升级这些"无聊但必要"的工作上。为什么不交给 AI？

### 核心展开

**第一层：冰山理论**
- 新代码只是冰山一角
- 每个交易、每次飞行都依赖多年前的代码
- Grit 专注水面下的维护工作

**第二层：GritQL 的威力**
- 不是文本匹配，是 AST 级操作
- Console.log 跨行也能识别
- 变量重定义（logger = console）也能追踪
- 编译器级别的语义理解

**第三层：AI 代理的自我修正**
- ReAct 框架：推理 → 行动 → 观察
- 递归修正循环：运行测试 → 失败 → 修改 → 再测试
- Healing step：最终通过才算成功

**第四层：分布对齐的智慧**
- Markdown：最适合 README 类内容
- Shell：最适合命令行操作
- 用模型见过的格式，降低理解成本

**第五层：真实世界的挑战**
- 遗留代码没有类型
- 遗留代码没有测试
- 解决方案：先生成测试，再作验证

### 收尾升华
**愿景**：让开发者从"无聊但必要"的维护中解放，专注于创造新价值。

**技术哲学**：
- 不要让 LLM 做复杂数学
- 给它一个"计算器"（GritQL）
- 专注于推理和决策

## 认知碰撞卡

### 卡片 1：冰山理论的商业机会

**对我意味着什么**：
当所有人都在追逐"新代码"工具时，90% 的"旧代码"市场被忽视了。这是典型的创业机会。

**反常识点**：
- Copilot 帮你写新代码，但那是少数
- 大部分代码是多年前的、需要维护的
- 维护工具的市场 > 新代码工具的市场

**可迁移场景**：
- 基础设施维护自动化
- 遗留系统现代化
- 任何" boring but necessary" 的工作

---

### 卡片 2：AST 级别的语义理解

**对我意味着什么**：
代码操作应该在语义级别，而不是文本级别。这是正确性的保证。

**反常识点**：
- 文本匹配会漏掉跨行的 console.log
- 文本匹配不认识变量重定义
- AST 操作保证语义正确性

**可迁移场景**：
- 任何代码转换工具
- 自动重构系统
- 代码搜索和分析

---

### 卡片 3：Healing Loop 的设计哲学

**对我意味着什么**：
AI 系统不应该一次性输出"正确"答案，而应该有自我修正的循环。失败是过程的一部分。

**反常识点**：
- 第一次版本不正确没关系
- 第 15 次版本可能正确
- 递归运行测试 → 修改 → 再测试

**可迁移场景**：
- 任何需要验证的 AI 生成
- 自动化测试修复
- 复杂任务的多步执行

---

### 卡片 4：分布对齐的工程智慧

**对我意味着什么**：
选择模型训练数据中最常见的格式作为接口，而不是自创格式。

**反常识点**：
- Markdown 对 README 最可靠
- Shell 对命令行最自然
- 用"分布"中的格式，降低理解成本

**可迁移场景**：
- Prompt 设计
- API 设计
- 任何 LLM 集成

---

### 卡片 5：测试即验证的务实策略

**对我意味着什么**：
当遗留代码没有测试时，先自动生成测试，作为修改的验证集。

**反常识点**：
- 自动生成的测试不完美，但比没有好
- 至少能保证修改不破坏现有行为
- 然后用这些测试验证 AI 的修改

**可迁移场景**：
- 任何遗留系统的现代化
- 重构前的安全网
- AI 生成代码的验证
