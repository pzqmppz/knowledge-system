# 知识体系操作准则 (SOP)

> 本文档定义知识体系的日常操作准则，确保即使丢失上下文也能快速有效执行

---

## 快速检查清单

当用户要求保存知识时，按以下步骤执行：

- [ ] 确认分类是否存在于 config.yaml
- [ ] 创建文件并添加 YAML Frontmatter
- [ ] 保存内容并添加反向链接区域
- [ ] 更新分类 INDEX.md
- [ ] 更新全局 INDEX.md 统计
- [ ] 更新 TAGS.md（如有新标签）
- [ ] 创建相关双向链接

---

## 1. 新增知识文件

### 1.1 确认分类

**检查文件**：`.knowledge/config.yaml` 中的 `categories` 列表

```yaml
categories:
  - name: "前端开发"
  - name: "后端开发"
  - name: "数据库"
  - name: "算法与数据结构"
  - name: "系统设计"
  - name: "DevOps运维"
  - name: "计算机基础"
  - name: "AI工具"
  - name: "工具与效率"
  - name: "软技能"
  - name: "其他"
```

**如果分类不存在**：先在 config.yaml 中添加分类定义

### 1.2 创建文件模板

**必填字段**：

```yaml
---
title: "知识标题"
category: "分类名称"
date: YYYY-MM-DD
tags: ["标签1", "标签2"]
---

# 文档标题

> 来源：（如有）
> 日期：YYYY-MM-DD
> 分类：

---

## 内容正文

---

## 🔗 反向链接

### 被引用
- 暂无引用

### 相关内容
- [[相关文件]] - 说明

### 待探索
- [[待学习文件]] - 说明

---
```

**可选字段**：

```yaml
---
source: "来源链接或描述"
author: "作者"
status: "draft|complete|updating|archived"
difficulty: "beginner|intermediate|advanced|expert"
priority: "low|medium|high|critical"
related: ["相关文件1.md", "相关文件2.md"]
---
```

### 1.3 文件命名规则

```
{描述性标题}.md
```

**示例**：
- `OpenClaw-Agent配置指南.md`
- `快速排序算法.md`
- `React状态管理方案对比.md`

**避免**：
- 过于简短：`笔记.md`
- 包含特殊字符：`笔记/01.md`
- 使用日期前缀：`2026-02-24笔记.md`

## 2. 双向链接规范

### 2.1 基本语法

```markdown
[[文件名]]                    # 基本链接
[[文件名|显示文本]]            # 带别名
[[文件名#章节标题]]            # 链接到章节
[[../分类/文件名]]             # 跨分类链接
![[文件名]]                    # 嵌入整个文件
![[文件名#章节]]               # 嵌入章节
```

### 2.2 必须创建链接的场景

| 场景 | 操作 |
|------|------|
| 提到另一个知识点 | 使用 [[双向链接]] |
| 前置知识要求 | 在文档开头添加 **前置知识**：[[...]] |
| 相关内容对比 | 添加 **相关内容**：[[...]] |
| 实际应用案例 | 添加 **应用场景**：[[...]] |
| 深入阅读推荐 | 添加 **扩展阅读**：[[...]] |

### 2.3 反向链接区域模板

每个文件末尾必须包含：

```markdown
---

## 🔗 反向链接

### 被引用
- [[引用的文件]] - 在哪个章节被引用

### 相关内容
- [[相关文件1]] - 关系说明
- [[相关文件2]] - 关系说明

### 待探索
- [[待学习文件]] - 为什么待学习

---
```

## 3. 索引更新规范

### 3.1 更新分类 INDEX.md

**位置**：`{分类}/INDEX.md`

**更新内容**：
1. 在文件列表中添加新文件条目
2. 更新标签索引
3. 更新最近更新时间

**模板**：

```markdown
## 📄 文件列表

### 文章与教程

| 文件 | 标签 | 难度 | 日期 |
|------|------|------|------|
| [文件名](./文件名.md) | #tag1 #tag2 | intermediate | 2026-02-24 |
```

### 3.2 更新全局 INDEX.md

**位置**：`INDEX.md`

**更新内容**：
1. 更新统计概览中的文件数
2. 更新最近更新表格
3. 如有新标签，更新标签云

### 3.3 更新 TAGS.md

**位置**：`TAGS.md`

**更新内容**：
1. 按字母顺序添加新标签
2. 更新标签的文件计数

## 4. 处理外部内容

### 4.1 从链接提取内容

**流程**：

1. 使用 webReader 或 webSearchPrime 获取内容
2. 提取核心信息，去除冗余
3. 翻译（如果是外文内容）
4. 按模板保存

**元数据必须包含**：

```yaml
---
source: "原始链接"
author: "原作者（如有）"
tags: ["#article", "..."]  # 添加来源类型标签
---
```

### 4.2 内容处理原则

| 原则 | 说明 |
|------|------|
| 结构化优先 | 保留原始结构，使用标题、列表、表格 |
| 去除冗余 | 删除广告、导航、重复内容 |
| 添加说明 | 在文档开头说明来源和处理日期 |
| 保留链接 | 重要的外部链接保留在参考资源章节 |

## 5. 定期维护任务

### 5.1 每次操作后检查

- [ ] 文件元数据是否完整
- [ ] 双向链接是否有效
- [ ] 索引是否已更新
- [ ] 标签是否一致

### 5.2 每周检查

- [ ] 检查失效链接
- [ ] 更新过时内容
- [ ] 合并重复笔记
- [ ] 审查 orphan 文件（无链接的文件）

### 5.3 每月检查

- [ ] 审查分类结构是否需要调整
- [ ] 检查 config.yaml 是否与实际一致
- [ ] 更新 README.md 内容
- [ ] 备份知识体系

## 6. 配置文件同步检查

### 6.1 检查点

**config.yaml vs 实际目录**：

```
检查项目：
- [ ] categories 列表与实际目录是否一致
- [ ] 新增分类是否已添加描述
- [ ] 忽略目录配置是否正确
```

**README.md vs 实际结构**：

```
检查项目：
- [ ] 目录结构说明是否准确
- [ ] 配置文件链接是否有效
- [ ] Obsidian 使用说明是否最新
- [ ] 统计数据是否正确
```

### 6.2 更新流程

当发现配置不同步时：

1. 确定哪个是权威来源（通常是实际目录结构）
2. 更新配置文件使其一致
3. 在相关文件的元数据中记录更新日期

---

## 7. 故障排除

### 7.1 Obsidian 打开失败

**解决步骤**：

1. 删除 `.obsidian/workspace.json`
2. 重新打开 Obsidian
3. 让 Obsidian 重新生成配置

### 7.2 链接失效

**类型** | **解决方法**
---|---
文件被移动 | 更新所有引用该文件的链接
文件被删除 | 从反向链接区域移除引用
文件重命名 | 更新所有 [[链接]]

### 7.3 元数据解析失败

**检查项目**：

- [ ] YAML 格式是否正确（注意缩进）
- [ ] 三个短横线 `---` 是否存在
- [ ] 日期格式是否为 `YYYY-MM-DD`

---

## 8. 用户指令快速响应

当用户说类似以下指令时：

| 用户指令 | 执行操作 |
|----------|----------|
| "帮我保存这个知识点" | 创建文件，添加元数据，更新索引 |
| "读取链接并保存" | 获取内容，翻译，按模板保存 |
| "更新索引" | 更新 INDEX.md 和 TAGS.md |
| "检查配置" | 对比 config.yaml 与实际结构 |
| "搜索关于 X 的内容" | 在对应分类中搜索 |
